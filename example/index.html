Getting Started Installation and Setup

Dependencies

The distribution already contains a very small subset of needed bootstrap styles. So the only dependency is AngularJS.

Bower

The easiest way to install angular-sAlertlocally is with bower:

bower install angular-sAlert

You can add the flag -Sto save the dependency in your bower.jsonfile.

Manual download

Alternatively you can download a release and copy the files located in the distfolder to your project.

Include files in your project

In order for angular-sAlertto work you need to include the following files in your project.

JavaScript:

<script type="text/javascript" src="angular-sAlert/dist/sAlert.js"></script>

CSS:

<link href="/bower_components/angular-sAlert/dist/sAlert.css" rel="stylesheet">

Installation

To use angular-sAlertyou need to integrate it in your angular application.

Define the sAlert angular module as a dependency:

angular.module('myApp', ['sAlert']);

To display the alerts you need to include the sAlert directive in your html:

<s-alert></s-alert>

To use angular-sAlertyou need to inject the sAlertservice via the angular dependency injection.

After that you can use the service to create and display new alerts. The sAlertservice proced a function for each alert type. Call the function of the alert type you want to be displayed and pass the alert message as a parameter.

There are three supported types of alert messages:

success 

sAlert.success('my success message');

info 

sAlert.info('my info message');

error 

sAlert.error('my error message');

Auto Removing Alerts open alerts that will remove themself after a while

The standard alerts need to be closed manually one by one by clicking the close icon on the right. But you can specify that an alert should close itself automatically after a while. The default timeout for the close is two seconds.

In order to add the auto removing behavior to an alert just append the following function to the declaration:

.autoRemove()

As an example, here is the definition of an auto removing success alert:

sAlert.success('i will remove myself soon').autoRemove();

This works with all types of alerts.

Alerts with promises close an alert once a promise is resolved

open alerts bound to a promise:
close alerts by resolving the promise

Alerts can also be connected to a promise. Once the promise is resolved or rejected the alert will be removed. This way loading indicators can be realized as a example use case.

To specify a promise for an alert append the following function to the alert definition:

.removeOnResolve(<promise>);

The full definition for an alert might be something like that:

sAlert.info('loading data...').removeOnResolve(myPromise);

Like autoRemovethis works for all types of alerts. In Angular you can use the $qmodule for manual promises or you can even use the promise returned by the $http module to indicate the status of a request as an example.

Configuration customize the display of alerts with various options

Options:

affects examples above

Open alerts:

You can configure various display options via the sAlertdirective. Just put an object in the optionsparam.

<s-alert options="{container: true}"></s-alert>

Alernativly you can directly add a scope object to the param. Be aware that the directive uses one-way-binding like ng-class, so you need to put your scope variable in double curly braces:

<s-alert options=""></s-alert>

Available options:

  • container (default: false): specify if alerts should be wrapped into a bootstrap container, making it smaller in width
  • fixedOnTop (default: true): specify if alerts should be fixed on top of the page so that they are always visible

The fixedOnTopoption allows you to display alerts locally on a page and by using multiple sAlertdirectives you can display alerts in multiple locations in your app. But by default all directives will display the same alerts. To have complete separate alerts you can connect a sAlertdirective to an instance:

<s-alert options="{fixedOnTop: false}" instance="myAlertInstance"></s-alert>

Set the option fixedOnTopto false to let the alerts be displayed in different locations.

Each alert definition can now be bound to an instance making it only visible in this location.Just add the name of the instance as a second parameter when declaring an alert:

sAlert.success('my success message', 'myAlertInstance');

Contribution development of the project

Install development dependencies

For the client side dependencies run bower:

bower install

The development workflow requires additional npm dependencies:

npm install

Unit tests

Karma is used as a unit test runner for AngularJS. The configs for the test runner are specified in karma.conf.js.

Run the unit tests:

karma start

A coverage report is generated in the folder coverage.

Automatic workflow

For automation angular-sAlertuses the task runner grunt. All tasks are defined in Gruntfile.js.

The default task runs unit tests, jshint, packages all files and copies them to the distfolder. To execute it run:

grunt

While developing you can let grunt watch for file changes and automatically run the default task on a change:

grunt watch

Versioning

Releases are versioned as follows:

<major>.<minor>.<patch>

Branching

The project uses git flow. New code is first implemented in feature branches and once stable merged to develop. The masterbranch contains stable releases each tagged with their version number.
